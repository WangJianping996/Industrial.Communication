# 通用工业通信类库七天开发计划

## 1. 项目目标

在 7 天内完成一个可发布到 NuGet 的通用工业通信类库 MVP，建立稳定、可扩展的公共 API，并实现常用传输通道、工业协议、设备适配器、报文监控及模拟测试能力。

7 天版本应优先保证以下内容：

- 传输通道、协议编解码、设备模型和业务调用彼此分离。
- 所有 I/O API 以异步方式为主，并支持超时与 `CancellationToken`。
- Serial Port、TCP Client/Server、UDP 具备可用实现。
- Modbus TCP/RTU 具备可用实现。
- Siemens S7、Mitsubishi MC、OPC UA 形成统一抽象，并至少完成约定范围内的读写闭环。
- 数字量 IO、运动控制卡、扫码器、称重设备能够通过适配器扩展。
- 具备重试、心跳、自动重连、状态通知、队列、粘包拆包、校验、报文监控、保存、导出、回放、地址映射、变量监视和模拟器的基础能力。
- 完成单元测试、集成测试、示例、API 文档和 NuGet 包配置。

> 注意：7 天适合交付“可扩展且可演示的 MVP”，不适合承诺覆盖每种 PLC 型号、全部协议指令和所有厂商运动控制卡。首版应明确支持矩阵，未覆盖的厂商能力通过扩展包逐步增加。

## 2. 建议技术基线

- 语言：C#，启用 Nullable、隐式 using 和 XML 文档。
- 目标框架：发布 `netstandard2.1`，使用现代 .NET 运行时执行测试，并验证串口及第三方依赖兼容性。
- 测试：xUnit、FluentAssertions、必要时使用 Testcontainers 或本地模拟服务。
- 日志：仅依赖 `Microsoft.Extensions.Logging.Abstractions`，不绑定 Serilog/NLog 等具体实现。
- 配置与依赖注入：使用 `Microsoft.Extensions.Options` 和 `Microsoft.Extensions.DependencyInjection.Abstractions`，同时允许不使用 DI 直接构造对象。
- 时间与重试：核心库避免强绑定重量级框架；重试策略通过接口或轻量实现提供。
- 版本策略：语义化版本；首个 NuGet 版本建议为 `0.1.0-preview.1`。
- 许可证：建议 MIT 或 Apache-2.0；引入第三方协议库前必须检查许可证、活跃度、目标框架和商业使用限制。

## 3. 建议解决方案结构

```text
Communication.sln
├─ src/
│  ├─ Communication.Abstractions/       # 公共接口、模型、异常、状态和事件
│  ├─ Communication.Core/               # 队列、重试、重连、心跳、监控、回放
│  ├─ Communication.Transports/          # Serial、TCP Client/Server、UDP
│  ├─ Communication.Protocols.Modbus/    # Modbus TCP/RTU
│  ├─ Communication.Protocols.S7/        # Siemens S7
│  ├─ Communication.Protocols.Mc/        # Mitsubishi MC
│  ├─ Communication.Protocols.OpcUa/     # OPC UA
│  ├─ Communication.Adapters/            # IO、运动卡、扫码器、称重设备通用适配器
│  └─ Communication.DependencyInjection/ # 可选的 DI 注册扩展
├─ tests/
│  ├─ Communication.UnitTests/
│  ├─ Communication.IntegrationTests/
│  └─ Communication.ProtocolTests/
├─ samples/
│  ├─ BasicClient/
│  ├─ ServerAndSimulator/
│  └─ PlcVariableMonitor/
├─ docs/
│  ├─ architecture.md
│  ├─ supported-features.md
│  ├─ getting-started.md
│  └─ protocol-guides/
└─ Directory.Build.props
```

可以按协议分别发布 NuGet 包，避免用户为了使用 TCP 而被迫引用 OPC UA 等大型依赖。推荐包关系为：`Abstractions <- Core <- Transports <- 各协议包/适配器包`，禁止反向依赖和协议包之间互相引用。

## 4. 核心抽象建议

首日应先冻结最小公共契约，后续实现全部围绕契约展开：

- `ITransportChannel`：`ConnectAsync`、`DisconnectAsync`、`SendAsync`、接收数据流、当前连接状态。
- `IProtocolCodec<TRequest,TResponse>`：编码请求、增量解码响应，不直接操作 Socket 或串口。
- `IFrameDecoder`：根据长度、分隔符、固定帧或自定义规则处理粘包与拆包。
- `IChecksum`：CRC16、LRC、累加和及用户自定义校验。
- `ICommunicationClient<TRequest,TResponse>`：组织发送、响应关联、超时、取消和重试。
- `ICommunicationServer`：监听、会话管理、客户端状态和请求分发。
- `IReconnectPolicy`、`IRetryPolicy`、`IHeartbeatStrategy`：策略可替换。
- `IMessageQueue<T>`：有界发送/接收队列、背压策略和关闭语义。
- `IMessageMonitor`、`IMessageStore`、`IMessageExporter`、`IMessageReplayService`：监控、历史、导出和回放。
- `IMessageRedactor`：按协议、方向、地址范围或字节区间脱敏；允许彻底禁止报文记录。
- `IPlcClient`：按地址读写单值或批量值。
- `IAddressParser`、`IValueConverter`：PLC 地址解析、大小端、字节序和类型转换。
- `IVariableMonitor`：变量轮询、变化通知、质量码、时间戳和失败状态。
- `IDeviceAdapter`：设备适配器生命周期和健康状态。
- `ICommunicationSimulator`：脚本响应、延迟、丢包、断线、错误帧和异常注入。

公共模型至少包括 `ConnectionState`、`ConnectionStateChangedEventArgs`、`MessageEnvelope`、`MessageDirection`、`CommunicationResult<T>`、`CommunicationError`、`RetryContext`、`VariableDefinition`、`VariableValue` 和 `DeviceHealth`。

## 5. 七天开发安排

### 第 1 天：需求边界、架构与公共契约

**上午**

- 建立解决方案、项目结构、代码规范、Nullable、XML 文档和统一构建配置。
- 明确首版支持矩阵：框架版本、操作系统、PLC 系列、协议版本、功能码、数据类型和已知限制。
- 调研 Siemens S7、Mitsubishi MC、OPC UA、Modbus 的第三方依赖，记录许可证与是否需要拆分独立 NuGet 包。
- 确定命名空间、包名、程序集名和版本规则，检查 NuGet 包 ID 是否被占用。

**下午**

- 实现 `Abstractions` 中的通道、协议、帧解析、校验、客户端、服务端、策略、监控、PLC 和适配器接口。
- 定义连接状态机：`Disconnected -> Connecting -> Connected -> Reconnecting -> Disconnecting/Faulted`。
- 统一异常与结果模型，区分超时、取消、连接失败、协议异常、校验失败和设备返回错误。
- 编写 `architecture.md` 和首批契约测试，防止协议层依赖具体传输类型。

**当日交付物**

- 可编译的解决方案骨架。
- 核心接口、模型、异常和状态机。
- 架构文档及依赖/许可证决策记录。

**验收标准**

- `dotnet build` 成功且无警告。
- 协议编解码器可以脱离网络和串口独立测试。
- 核心抽象不出现 `Socket`、`SerialPort` 或具体 PLC SDK 类型泄漏。

### 第 2 天：传输通道、队列与帧处理

**上午**

- 实现 Serial Port 通道：波特率、数据位、停止位、校验位、读写超时和端口资源释放。
- 实现 TCP Client：连接、断开、异步收发、半开连接处理、超时与取消。
- 实现 TCP Server：监听、多个会话、会话 ID、并发上限和优雅停止。
- 实现 UDP：绑定、本地/远端端点、单播收发，并预留广播和组播配置。

**下午**

- 使用 `Channel<T>` 或等价结构实现有界发送队列和接收队列，定义等待、拒绝或丢弃等背压策略。
- 实现固定长度、长度字段、分隔符、静默间隔和自定义帧解码器，覆盖粘包、半包和多帧连包。
- 实现 CRC16、LRC 和自定义 `IChecksum` 扩展点。
- 补充并发发送、断开时清理、资源释放、慢消费者和大报文测试。

**当日交付物**

- Serial、TCP Client/Server、UDP 通道实现。
- 收发队列、帧解码和校验模块。
- 基础 Echo Server 示例。

**验收标准**

- 同一协议编解码器可以切换不同通道而无需修改代码。
- 测试覆盖随机分片、多个帧一次到达、错误校验和取消等待。
- 队列达到容量上限时行为明确，不出现无限内存增长。

### 第 3 天：可靠通信与可观测性

**上午**

- 实现请求/响应关联、单请求串行模式和支持事务 ID 的并发模式。
- 实现统一超时、`CancellationToken`、可配置重试、指数退避与随机抖动。
- 实现心跳策略、自动重连和最大重连次数；主动断开后禁止自动重连。
- 实现连接状态通知、线程安全状态转换和重连后的订阅/监视恢复钩子。

**下午**

- 实时报文监控：方向、通道 ID、会话 ID、协议、时间戳、耗时、原始字节和解析摘要。
- 历史存储接口与首个文件实现，支持滚动文件、容量限制、保留天数和查询过滤。
- 实现 CSV/JSON 导出和按时间、方向、连接、协议筛选。
- 实现报文回放，支持原始时间间隔、倍速、固定间隔和仅发送指定方向。
- 实现 `IMessageRedactor`，默认关闭敏感载荷记录；允许按字节区间或解析字段掩码，并记录“已脱敏”标志。

**当日交付物**

- 超时、取消、重试、心跳、自动重连和状态通知。
- 监控、历史、导出、回放和脱敏基础实现。

**验收标准**

- 模拟断网后可以自动恢复，用户主动断开不会再次连接。
- 取消操作能及时退出且不残留待处理请求。
- 默认日志不输出凭据、令牌或完整敏感报文。
- 历史存储失败不能阻塞或中断主通信链路。

### 第 4 天：Modbus TCP/RTU 与通信模拟器

**上午**

- 实现 Modbus TCP 的 MBAP 编解码、事务 ID、单元 ID 和异常响应。
- 实现 Modbus RTU 的站号、功能码、CRC16、串口静默间隔和请求串行化。
- 首版至少覆盖功能码 01、02、03、04、05、06、15、16；若时间允许增加 22、23。
- 实现线圈、离散输入、保持寄存器、输入寄存器的单点与批量读写。
- 明确寄存器地址基准，避免 0 基地址与 4xxxx 展示地址混用。

**下午**

- 实现 Modbus TCP/RTU 模拟从站，可配置寄存器、响应延迟、异常码、超时、断线和 CRC 错误。
- 为模拟器增加脚本式响应接口，供其他协议测试复用。
- 使用公开协议示例或抓包样本建立 Golden Tests，验证编码字节完全一致。
- 编写 Modbus 快速入门和 TCP/RTU 示例。

**当日交付物**

- 可用的 Modbus TCP/RTU 客户端及基础模拟从站。
- 协议字节级测试和示例代码。

**验收标准**

- TCP 与 RTU 共享数据模型和功能码逻辑，只替换帧封装与通道。
- 正常、异常响应、CRC 错误、超时、粘包和拆包均有测试。
- 模拟从站可以支撑无需真实硬件的 CI 集成测试。

### 第 5 天：Siemens S7、Mitsubishi MC 与 PLC 变量体系

**上午**

- 实现统一 `IPlcClient`、地址模型、批量读写和类型转换。
- Siemens S7 首版限定并记录支持范围，例如 S7-1200/1500、ISO-on-TCP、DB/I/Q/M 区、Bit/Byte/Int/DInt/Real/String。
- Mitsubishi MC 首版限定并记录支持范围，例如 3E Frame Binary、TCP、X/Y/M/D/W 等常用软元件。
- 如使用第三方库，以内部适配层隔离其类型，避免第三方类型进入公共 API。

**下午**

- 实现 PLC 地址映射：变量名、协议地址、数据类型、长度、读写权限、缩放系数和描述。
- 实现变量监视：轮询周期、批量合并、值变化事件、质量码、时间戳、单变量失败隔离。
- 实现大小端、字节交换、字交换和字符串编码策略。
- 增加协议帧 Golden Tests；真实 PLC 不可用时，至少使用模拟响应完成读写闭环。

**当日交付物**

- S7 与 MC 独立扩展包的首版读写能力。
- PLC 地址映射、变量监视与类型转换模块。
- 明确的 PLC 型号/地址/类型支持矩阵。

**验收标准**

- 用户通过统一变量定义即可监视 Modbus、S7 或 MC 变量。
- 批量读取能按协议限制自动分组，单个变量异常不导致整个监视服务停止。
- 第三方协议依赖可以在未来替换，不破坏核心公共 API。

### 第 6 天：OPC UA 与通用设备适配器

**上午**

- 使用 OPC Foundation 官方 .NET Standard SDK 或经许可证审查的实现创建独立 OPC UA 包。
- 实现端点发现、会话连接、用户名/密码或证书认证、读写 Node、订阅和自动恢复。
- 暴露安全策略、消息安全模式、证书校验和信任库配置；默认配置不得静默信任所有证书。
- 将 OPC UA 订阅值转换为统一的变量值、质量码和时间戳模型。

**下午**

- 实现适配器基类，组合通道、帧解码器和命令解析器。
- 数字量 IO：读取输入、设置输出、批量状态和边沿变化事件。
- 运动控制卡：轴状态、使能、回零、绝对/相对运动、停止；危险动作必须显式调用，不允许重连后自动重放。
- 扫码器：分隔符/长度帧、扫码结果、重复过滤和触发命令。
- 称重设备：稳定值、毛重/净重、单位、去皮/清零命令和连续上报解析。
- 提供通用 Lambda/Delegate 适配器，使用户可快速接入私有 ASCII 或二进制设备。

**当日交付物**

- OPC UA 读写、订阅和安全配置示例。
- IO、运动控制卡、扫码器、称重设备的通用契约与参考适配器。

**验收标准**

- OPC UA 断线恢复后能够重建订阅。
- 不安全证书配置需要用户显式启用并产生警告。
- 每类设备至少有一个模拟器或伪设备测试，且厂商 SDK 类型不进入通用接口。

### 第 7 天：质量收口、文档、打包与发布演练

**上午**

- 完成单元测试、集成测试、并发测试、长时间运行测试和失败注入测试。
- 重点检查连接/断开竞争、重连风暴、队列泄漏、Socket/串口释放、事件订阅泄漏和线程安全。
- 使用公开 API 审查工具检查破坏性变更，补齐 XML 文档和 Nullable 标注。
- 启用 Source Link、确定性构建、符号包 `.snupkg`、README、LICENSE 和包图标。

**下午**

- 编写 `getting-started.md`、各协议示例、支持矩阵、故障排查和安全说明。
- 配置 CI：restore、build、test、pack；Tag 触发正式发布，普通分支只生成预览包或构建产物。
- 在本地 NuGet 源安装全部包，运行示例，检查依赖是否最小化、README 是否正常展示。
- 先发布 `0.1.0-preview.1`，验证包内容后再决定是否推送稳定版。
- 创建后续 Backlog：更多 S7/MC 型号、OPC UA 高级特性、数据库历史存储、更多厂商适配器和性能优化。

**当日交付物**

- 可安装的 `.nupkg` 和 `.snupkg`。
- 完整 README、API 文档、示例和支持矩阵。
- CI/CD 配置和发布检查清单。

**验收标准**

- 全新项目只引用 NuGet 包即可运行示例，不依赖源码目录中的隐式文件。
- Release 构建无警告，全部自动化测试通过。
- 包内不包含测试数据、密钥、证书私钥、真实设备地址或敏感报文。
- NuGet 包元数据、许可证、仓库地址、版本号和依赖范围正确。

## 6. 每日固定节奏

每天除功能开发外，固定保留以下时间：

- 09:00-09:20：确认当天范围，禁止临时扩大协议支持面。
- 12:00 前：完成核心接口或最小闭环，避免下午才首次集成。
- 16:30-17:30：补测试、示例和文档。
- 17:30-18:00：运行完整构建与测试，记录次日风险。

每日完成定义：代码已合并、测试已通过、文档已更新、示例可运行，而不是仅“功能代码已写完”。

## 7. 测试矩阵

| 类别 | 必测场景 |
|---|---|
| 生命周期 | 重复连接、重复断开、连接中取消、发送中断开、资源释放 |
| 超时与取消 | 连接超时、发送超时、响应超时、外部取消、取消与响应竞态 |
| 重试与重连 | 瞬时失败、持续失败、指数退避、主动断开、恢复后的状态与订阅 |
| 帧处理 | 半包、粘包、多帧、空帧、超长帧、错误长度、错误校验 |
| 队列 | 并发生产、慢消费、满队列、关闭队列、异常消费者 |
| 协议 | 正常响应、异常响应、非法地址、非法功能码、边界长度 |
| PLC 数据 | 位、整数、浮点、字符串、大小端、跨寄存器、批量分组 |
| 可观测性 | 实时监控、历史过滤、滚动清理、导出、倍速回放 |
| 安全 | 默认不记录载荷、字段脱敏、证书校验、日志注入、路径安全 |
| 稳定性 | 长时间运行、频繁重连、高并发、多客户端、内存与句柄增长 |

建议为解析器和编解码器加入模糊测试或随机分片测试，因为这部分最容易在真实网络流量下暴露边界问题。

## 9. 首版完成定义

满足以下条件时，首个预览版才算完成：

1. Serial、TCP Client/Server、UDP 均可通过示例完成收发。
2. Modbus TCP/RTU 可完成常用线圈和寄存器读写。
3. S7、MC、OPC UA 在已声明的支持范围内完成至少一次读写或订阅闭环。
4. IO、运动控制、扫码器、称重设备具有稳定通用接口和可运行参考适配器或模拟器。
5. 全链路支持超时、取消、重试、心跳、自动重连和状态通知。
6. 粘包、拆包、校验、队列及异常响应均有自动化测试。
7. 报文可实时查看、查询历史、导出和回放，并支持脱敏或禁止记录。
8. PLC 地址映射和变量监视可跨至少两种 PLC 协议工作。
9. CI 能独立完成构建、测试与打包，新项目可从本地 NuGet 包运行示例。
10. README、API 文档、支持矩阵、许可证和安全说明完整。

## 10. 第 7 天之后的优先级

首个预览版发布后，建议按以下顺序迭代：

1. 收集真实设备互操作结果，修复协议兼容性和资源泄漏问题。
2. 扩展 S7、MC 的设备系列、地址类型和高级指令。
3. 为常见 IO、运动控制卡、扫码器和称重仪表增加独立厂商扩展包。
4. 增加 SQLite/数据库报文历史存储及索引查询。
5. 增加性能基准、链路指标、OpenTelemetry 集成和诊断工具。
6. API 稳定后发布 `1.0.0`，并制定兼容性与弃用策略。
